# business-spec-to-golden

一个 Claude Code Skill，用于将需求文档转换为详细设计规格和黄金参考程序，采用测试驱动开发。

## 概述

本 Skill 将模糊的需求转换为：

1. **文档理解笔记** - 将对源文档的理解外化保存
2. **问题清单** - 基于文件的 P0/P1/P2 分层问题及答案
3. **分层中间件输出** - 全局流程、子模块设计、边界规则
4. **经过评审的设计规格** - 可直接用于实现的设计文档
5. **测试用例** - 带 CLI 入口，可验证程序正确性
6. **黄金参考程序** - 通过所有测试的参考实现

## 核心特性

- **文件优先规则** - 每个问题必须从问题清单文件读取，绝不能凭记忆提问
- **P0/P1/P2 分层** - 全局 → 子模块 → 边界，严格按顺序提问
- **中间件输出** - 每层收敛后必须输出结构化的 markdown 文件
- **测试驱动开发** - 黄金程序必须通过所有测试用例
- **自动化规格评审** - 子代理在用户评审前先评审设计文档

## 两个版本

| 版本 | 文件 | 行数 | 目标模型 |
|------|------|------|----------|
| **通用版** | `SKILL.md` | ~230 | 所有模型 |
| **完整版** | `SKILL-FULL.md` | ~350 | 强模型（Claude, GPT-4） |

### 版本差异

| 特性 | SKILL.md | SKILL-FULL.md |
|------|----------|---------------|
| 工作流步骤详细程度 | 基础 | 详细 |
| 测试文件示例 | 简略 | 完整示例 |
| 错误恢复表格 | 4 项 | 5 项 |
| 状态跟踪 | 简单 | 详细 |

## 工作流（10 个阶段）

```
1. 文档理解 → *-understanding.md
2. 问题清单 → *-questions.md (P0/P1/P2)
3. P0 澄清 → *-global-flow.md
4. P1 澄清 → *-submodule-design.md
5. P2 澄清 → *-boundary-rules.md
6. 设计文档 → *-design.md
7. 规格评审（子代理）
8. 用户评审确认
9. 测试用例 → *-test.js（带 CLI）
10. 黄金程序 → 测试驱动迭代
```

## P0/P1/P2 分层定义

| 层级 | 范围 | 问题类型 |
|------|------|----------|
| **P0** | 全局 | 输入输出格式、I/O 契约、子模块列表、执行顺序、模块间数据流 |
| **P1** | 子模块 | 每个模块的输入/输出、转换逻辑、决策规则 |
| **P2** | 边界 | 空/无效输入、边界值、异常处理 |

## 输出文件

所有文件保存在 `docs/business-specs/YYYY-MM-DD-<topic>-*`：

| 文件 | 内容 |
|------|------|
| `*-understanding.md` | 文档结构、模块清单、已知 I/O |
| `*-questions.md` | P0/P1/P2 问题及状态和答案 |
| `*-global-flow.md` | 流程图、I/O 契约、模块关系 |
| `*-submodule-design.md` | 每个子模块的输入/输出/转换 |
| `*-boundary-rules.md` | 边界规则、异常处理 |
| `*-design.md` | 完整实现规格 |
| `*-test.js` | 测试用例 + CLI（`node *-test.js --all`） |

## 使用方法

```
/business-spec-to-golden
```

或直接提供需求文档：

```
这是我的产品需求文档 [附上文档]。
请帮我生成设计文档和参考实现。
```

## 测试驱动的黄金程序

黄金程序采用测试优先的开发方式：

1. 收集用户提供的输入/输出示例
2. 设计边界和异常测试用例
3. 编写带 CLI 入口的测试文件
4. 实现黄金程序
5. 迭代直到所有测试通过

```bash
# 运行所有测试
node docs/business-specs/2024-01-15-promo-test.js --all
```

## 示例会话

**输入**：促销规则文档

**过程**：
1. Agent 编写理解笔记
2. Agent 构建问题清单（P0: 5个, P1: 8个, P2: 4个）
3. P0：询问输入、输出、模块 → 编写 global-flow.md
4. P1：询问每个模块的逻辑 → 编写 submodule-design.md
5. P2：询问边界情况 → 编写 boundary-rules.md
6. 编写 design.md
7. 子代理评审设计
8. 用户确认
9. 用户提供测试示例，Agent 编写测试文件
10. Agent 实现黄金程序，所有测试通过
