# 评分标准 - business-spec-to-golden (v2)

## 评估理念

**过程规则是手段，产出质量是目的。**

本评分标准聚焦于：agent 是否通过澄清流程，真正理解了用户诉求，并产出了准确的设计文档和正确的 golden 程序。

---

## 核心评分维度（决定最终得分）

### 1. 歧义识别完整性 (ambiguity_coverage)

Agent 是否识别出了需求文档中的关键歧义？

| 分数 | 描述 |
|------|------|
| 0 | 没有识别任何歧义，直接开始写设计 |
| 1 | 识别了少量歧义，但遗漏了重要问题 |
| 2 | 识别了大部分歧义，有少量遗漏 |
| 3 | 识别了所有关键歧义，包括隐含问题 |

**测试方法**：测试用例中预设已知的歧义点，检查是否被识别

### 2. 需求理解准确性 (understanding_accuracy)

通过澄清后，agent 是否正确理解了用户的业务意图？

| 分数 | 描述 |
|------|------|
| 0 | 理解错误，与用户意图相反 |
| 1 | 部分理解，但关键决策点理解错误 |
| 2 | 基本理解，有少量偏差 |
| 3 | 完全准确理解用户意图，包括边界情况 |

**测试方法**：对比用户真实意图与 agent 的业务逻辑总结

### 3. 设计文档质量 (design_doc_quality)

设计文档是否详细、准确、可实施？

| 分数 | 描述 |
|------|------|
| 0 | 设计文档缺失或与需求不符 |
| 1 | 有设计文档，但缺少关键章节或决策规则不清晰 |
| 2 | 设计文档完整，决策规则基本清晰，可实施性一般 |
| 3 | 9章节完整，决策规则精确，边界情况明确，两个工程师读后能产出相同实现 |

**检查点**：
- [ ] Objective 清晰定义业务目标
- [ ] Domain Model 准确定义实体和含义
- [ ] Input/Output Contract 完整
- [ ] Decision Rules 有明确的优先级
- [ ] Edge Cases 覆盖完整
- [ ] Examples 包含正常和边界情况

### 4. Golden 程序正确性 (golden_correctness)

Golden 程序是否正确实现了设计文档？

| 分数 | 描述 |
|------|------|
| 0 | 无法运行或与设计文档完全不符 |
| 1 | 可运行但逻辑错误，无法通过测试用例 |
| 2 | 基本正确，部分边界情况处理有误 |
| 3 | 完全正确，所有测试用例通过，边界情况处理完善 |

**测试方法**：使用设计文档中的 Examples 作为测试用例

### 5. Golden 程序可读性 (golden_readability)

Golden 程序是否易于审查和理解？

| 分数 | 描述 |
|------|------|
| 0 | 代码混乱，难以理解 |
| 1 | 基本可读，但缺少必要注释或结构混乱 |
| 2 | 结构清晰，有基本注释 |
| 3 | 代码自解释，关键逻辑有注释链接到设计文档 |

## 过程合规维度（用于诊断，不影响总分）

这些维度帮助诊断：如果产出质量不好，是哪个过程环节出了问题？

### P1. 单问题规则 (single_question)

Agent 是否每次只问一个问题？

| 分数 | 描述 |
|------|------|
| 0 | 经常批量提问 |
| 1 | 偶尔批量提问 |
| 2 | 大部分时候单问 |
| 3 | 严格单问 |

### P2. 业务聚焦 (business_focus)

问题是否聚焦业务而非工程？

| 分数 | 描述 |
|------|------|
| 0 | 问了大量工程问题 |
| 1 | 混杂工程问题 |
| 2 | 基本聚焦业务 |
| 3 | 严格业务聚焦 |

### P3. 门控遵守 (gate_compliance)

是否在每个阶段等待用户确认？

| 分数 | 描述 |
|------|------|
| 0 | 跳过所有门控 |
| 1 | 跳过部分门控 |
| 2 | 大部分门控遵守 |
| 3 | 所有门控都遵守 |
